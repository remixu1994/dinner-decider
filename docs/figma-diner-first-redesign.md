# 干饭人优先 Figma 产品稿说明

## 目标
- 把原来的“单页切模式工作台”重构成双视图。
- 干饭人首页只做三件事：表达想吃什么、查看今晚菜单、提交餐后反馈。
- 厨师拥有独立工作台，集中处理诉求汇总、推荐、选菜、确认和审核。
- 移动端优先，保证单手可用和首屏主任务明确。

## 信息架构
- `00 Product Map`
  - 角色：`DINER`、`CHEF`、`OWNER`、`PENDING_MEMBER`
  - 入口规则：`DINER -> /family`，`CHEF/OWNER -> /chef`
  - 状态流：`COLLECTING -> PLANNING -> PUBLISHED -> FEEDBACK -> COMPLETED`
- `01 User Flows`
  - 创建家庭
  - 输入家庭码加入
  - 干饭人两步提交
  - 厨师采用推荐并确认菜单
  - 成员逐菜反馈
- `02 Foundations`
  - 色板：暖米白底色、橙色强调、石色文字
  - 圆角：卡片 `28-32px`，按钮 `16-18px`
  - 间距：页面 `20-24px`，卡片内 `16-20px`
- `03 Components`
  - 顶部状态条
  - 菜品选择 Chip
  - 标签 Chip
  - 菜品卡片
  - 今日诉求卡
  - 反馈单选组
  - 底部主 CTA
- `10 Diner Mobile`
  - 干饭人首页空状态
  - 两步提交态
  - 已提交态
  - 已发布菜单态
  - 餐后反馈态
- `20 Chef Mobile`
  - 今日诉求汇总
  - 推荐方案
  - 今晚菜单编辑
  - 临时新增菜
  - 待审核成员
- `30 Shared Pages`
  - 创建/加入家庭
  - 历史菜单
  - 家庭设置
- `40 Desktop Adaptation`
  - 干饭人首页桌面适配
  - 厨师工作台桌面适配
- `90 Prototype`
  - 加入家庭 -> 干饭人提交 -> 厨师确认 -> 干饭人查看菜单 -> 餐后反馈

## 干饭人首页
- 首屏结构
  - 状态卡：今日状态、已提交人数、今晚菜单道数
  - 主模块：`今晚想吃什么`
  - 次模块：`家庭今日想法`、`今晚菜单`、`餐后反馈`
  - 底部：最近吃过
- 交互重点
  - 第 1 步先点选菜品、标签或“都可以”
  - 第 2 步可选补一句备注
  - 默认开启“接受系统推荐”
  - 菜单已发布后，提交区自动降级为只读状态说明

## 厨师工作台
- 首屏结构
  - 诉求汇总
  - 今晚菜单
  - 推荐方案
  - 家庭常做菜
  - 临时新增菜
  - 待审核成员
- 交互重点
  - 推荐方案支持整套采用
  - 当前菜单支持单道菜确认、移除
  - 当晚菜单至少一菜后才能发布
  - 审核入口不进入干饭人首页

## 组件说明
- `Selectable Chip`
  - 用于点选想吃的菜和标签
  - 未选中：浅底描边
  - 选中：橙底白字
- `Dish Card`
  - 标题、份量、标签、食材克数
  - 厨师版右侧带操作按钮
- `Feedback Group`
  - 一行一个维度：喜欢程度、咸淡、分量
  - 采用单选 pill，不出现下拉框

## 文案原则
- 干饭人用语更轻松直接，例如“今晚想吃什么”“都可以”“提交我的想法”
- 厨师用语强调决策，例如“整套采用”“确认今晚菜单”“加入今晚菜单”
- 所有提示都避免后台语言和字段化描述

## 原型验证点
- 干饭人在 15 秒内完成一次提交
- 干饭人不需要理解“模式切换”
- 厨师在一个工作台内完成从看诉求到发布菜单
- 已发布菜单回到干饭人首页时，优先展示最终菜单和反馈入口
